# Задача: бонусный пирог с капустой при заказе от 3000 ₽

> Запрос Рамиля (2026-06-16): когда сумма заказа больше 3000 ₽ — автоматически, чтобы
> клиенту было видно, что у него в заказе есть бонусный пирог с капустой, 1 кг.

## Что нужно

При сумме товаров > 3000 ₽ в заказ автоматически добавляется **бесплатный** пирог,
клиент видит его во всех местах, где видит состав заказа.

**Бонусный товар** (из `frontend-next/lib/menu.ts`):
- `id: osetinskiy-pirog-s-kapustoi-kabuskadzhyn` — «Осетинский пирог с капустой "Кабускаджын"»
- Вес **1000 г (1 кг)** (обычная цена 890 ₽), в заказе — цена **0 ₽**, пометка «Бонус».

## Поведение

1. subtotal > 3000 ₽ → бонус появляется автоматически.
2. subtotal ≤ 3000 ₽ → бонус автоматически исчезает.
3. Бонус не влияет на итог (0 ₽), его нельзя удалить вручную и нельзя менять кол-во (всегда 1 шт).
4. Виден везде: корзина (CartDrawer), оформление (checkout / OrderSummary), страница `/order`, письмо на почту.

## Открытые вопросы (уточнить у Рамиля или взять дефолт)

- **Порог по subtotal (стоимость пирогов) или total (с доставкой)?** Дефолт: по **subtotal**.
- **Один бонус на заказ или ещё один за каждые +3000 ₽?** Дефолт: **один**.
- Граница — строго `> 3000` или `>= 3000`? Дефолт: `> 3000` (как написал Рамиль «больше 3000»).

## План реализации

Ключевая идея: бонус — **виртуальный (производный)**, а не настоящий элемент стейта.
Не пишем его в `items`/localStorage, а вычисляем «на лету». Так не будет гонок
авто-добавления/удаления и мусора в хранилище. Всё централизуется в `useCart`.

### 1. `frontend-next/hooks/useCart.tsx` — ядро логики
- Завести константу порога и id бонуса: `BONUS_THRESHOLD = 3000`, `BONUS_PRODUCT_ID`, `BONUS_WEIGHT = '1000 г'`.
- Добавить в контекст вычисляемое `bonusItem: CartItem | null` (через `useMemo` от `subtotal`):
  - если `subtotal > BONUS_THRESHOLD` → собрать `CartItem` из меню с `price: 0`, `qty: 1`,
    флагом `isBonus: true` (добавить поле в интерфейс `CartItem`);
  - иначе `null`.
- Добавить `displayItems = bonusItem ? [...items, bonusItem] : items` — это то, что рендерят UI и шлёт checkout.
- `subtotal` оставить как есть (по реальным `items`), бонус его не меняет (он 0 ₽).
- В `updateQty`/`removeItem` — защититься от изменения бонуса (если `id` бонусный — игнор),
  на случай если кнопки всё же отрендерятся.

### 2. `frontend-next/components/CartDrawer.tsx`
- Рендерить `displayItems` вместо `items`.
- Для строки с `isBonus`: бейдж «Бонус», цена «0 ₽» / «Бесплатно», **скрыть** кнопки +/–/удалить.
- (Опц.) Подсказка-прогресс: «До бонусного пирога осталось N ₽», когда subtotal чуть ниже порога.

### 3. `frontend-next/components/checkout/OrderSummary.tsx`
- Принимать/рендерить `displayItems`; строку бонуса показать с пометкой и ценой 0 ₽.
- Счётчик «N позиций» — решить, считать ли бонус (скорее да, для наглядности).

### 4. `frontend-next/app/checkout/page.tsx`
- В payload заказа (`items: cartItems.map(...)`, строка ~111) слать `displayItems`,
  чтобы бонус попал в заказ и письмо. `subtotal/total` не трогаем — бонус 0 ₽.

### 5. `backend/server.js` — письмо `sendOrderEmail` (строки ~129-174)
- Шаблон уже рендерит `payload.items` с `price*qty`. Бонус с `price:0` отрисуется как «0 ₽».
- (Опц.) Если в item есть `isBonus` — дописать к названию «(бонус)» для ясности в письме.
- Бэк ничего не пересчитывает по сумме — бонус приходит готовым в payload. Перезалить бэк по инструкции из CLAUDE.md.

### 6. Проверка
- Корзина < 3000 → бонуса нет; добавить до > 3000 → бонус появился во всех местах; убрать товары → бонус исчез.
- Оформить заказ → бонус в письме и на `/order`, итоговая сумма не изменилась.
- `npm run build` + деплой фронта; при правке бэка — перезалить и `pm2 restart alanpir-api`.
